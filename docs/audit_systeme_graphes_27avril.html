⚠️ 4. Anomalies découvertes pendant l'audit

Anomalie #1 : graph_indexer cassé fonctionnellement
196 conversations ont été insérées comme nœuds dans semantic_graph_nodes mais avec 0 edge. La fonction _extract_entity_keys retourne des clés au format Système B (francine-coullet), alors qu'elle cherche dans le Système A qui utilise des clés au format odoo-partner-3795. Match impossible.
Conséquence : Quand l'utilisateur demande "qu'est-ce qu'on s'est dit sur Francine ?", aucune conversation passée n'est retrouvée.
Anomalie #2 : Tenant orphelin "couffrant" (sans _solar)
La DB contient un tenant_id couffrant avec :
- 114 712 nœuds dans semantic_graph_nodes (vs 5 644 pour couffrant_solar)
- 562 971 edges dans semantic_graph_edges (vs 6 007)
- 60 426 chunks dans odoo_semantic_content (vs 1 711)
C'est probablement un ancien tenant_id avant le renommage en couffrant_solar. Données fantômes jamais nettoyées. Pollution importante.
Anomalie #3 : Doublons silencieux entre A et B
Exemple Francine Coullet existe à 4 endroits différents :
- 2 nœuds Person dans Système A (odoo-partner-3795, odoo-partner-3643)
- 11 ressources sous coullet-francine dans Système B
- 1 ressource sous francinecoullet41@gmail.com dans Système B
- 6 nœuds Conversation dans Système A (conv_202, conv_337, conv_338, conv_340, conv_341, conv_342)
Aucune liaison entre les 4 silos.
Anomalie #4 : gmail_polling DÉSACTIVÉ par défaut
Le polling Gmail nécessite la variable d'env SCHEDULER_GMAIL_ENABLED=true. Probablement intentionnel (tu utilises Outlook), mais à vérifier que ce n'est pas un oubli si tu voulais ingérer les mails Gmail.
Anomalie #5 : populate_from_* sont des appels MANUELS
populate_from_mail_memory, populate_from_calendar, populate_from_contacts, populate_from_drive sont tous appelables uniquement via le panel admin (route /admin/system/populate). Aucun job cron ne les déclenche automatiquement.
Conséquence : Si tu ne cliques pas, le Système B (entity_links) ne se met pas à jour automatiquement après changements. Sauf populate_from_odoo qui peut être déclenché par le webhook.
Observation #6 : Les peupleurs Drive/Calendar/Mail nourrissent les DEUX systèmes
drive_scanner écrit dans Système A (graph) ET dans entity_links via populate_from_drive.
Mais cette double-écriture n'est pas synchrone : le drive_scanner peuple A en temps réel, et populate_from_drive peuple B uniquement sur demande manuelle.
Conséquence : les deux systèmes peuvent être désynchronisés à tout moment.

💡 5. Architectures cibles possibles

Option 1 — Fix minimal (court terme)

  • Réparer graph_indexer en faisant matcher par node_label dans Système A.
  • Effort : 30 min — Bénéfice cœur : ~zéro (le cœur ne lit pas A).
  • Bénéfice : les outils search Raya retrouveraient les conversations passées via traverse.

Option 2 — Re-router graph_indexer vers Système B (recommandé court terme)

  • graph_indexer insère désormais dans entity_links : entity_key = "coullet-francine", resource_type = "conversation", resource_id = conv_42.
  • Le cœur (qui lit entity_links) voit immédiatement les conversations passées dans le contexte d'un contact.
  • Effort : 45 min — Bénéfice cœur : élevé.
  • Inconvénient : on accentue la dépendance au Système B avant unification.

Option 3 — Unification A+B (moyen/long terme, ~1 semaine)

  • Créer une couche d'abstraction unique app.knowledge_graph qui accepte les deux types d'entrées (clé technique et clé humaine).
  • Migration progressive : ajouter une colonne entity_key_human dans semantic_graph_nodes pour pouvoir matcher par nom.
  • Réécriture des consommateurs pour passer par cette couche unique.
  • Suppression du système entity_links (devient une vue SQL).
  • Effort : 5-7 jours — Bénéfice : architecture saine, durable.
  • À faire après le renommage Saiyan et la stabilisation B.

Option 4 — Bonus pré-requis (à faire de toute façon)

  • Nettoyer le tenant orphelin couffrant : 738 109 lignes fantômes à purger (114k nœuds + 562k edges + 60k chunks). Pollution énorme.
  • Décider du sort du tenant : suppression totale OU migration des données vers couffrant_solar.
  • Effort : 30 min audit + 15 min purge — Bénéfice : performance DB (indexes plus petits) + clarté.
  • À faire avant de toucher au reste pour ne pas raisonner sur des données pourries.

📌 6. Recommandation finale

Plan d'action en 3 étapes ordonnées :

Étape 1 (ce week-end, 1h) : Nettoyer le tenant orphelin couffrant. Audit + décision (purge ou migrer) + exécution.

Étape 2 (ce week-end, 45 min) : Re-router graph_indexer vers Système B (Option 2). Bénéfice immédiat sur le cœur de Raya. Reset du flag indexed_in_graph pour ré-indexer les 196 conversations vers le bon endroit.

Étape 3 (semaine prochaine ou plus tard) : Plan d'unification A+B (Option 3). Refonte propre, durable.
